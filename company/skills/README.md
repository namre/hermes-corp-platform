# Корпоративные skills

Skills — главный рычаг кастомизации агента под конкретную компанию. Вместо
того чтобы прописывать корпоративный контекст в system prompt каждого runtime,
он выносится в отдельный файл `SKILL.md`, который монтируется в runtime
read-only. Агент читает его как часть своей инструкции при старте.

---

## Что такое корп-skill

Skill — это директория с файлом `SKILL.md`. Файл содержит:
- маппинг корпоративных сервисов (как называется таск-трекер, где лежат файлы);
- поведенческие правила (как отвечать, что цитировать, кого эскалировать);
- корпоративный контекст (название компании, подразделения, типовые термины).

Skill не содержит кода и не запускается как процесс. Это инструкция для LLM,
поставляемая как отдельный артефакт, а не зашитая в образ агента.

---

## Как монтируется в runtime

При создании user-runtime provisioner монтирует директорию skill(ов) как
read-only volume:

```yaml
# Фрагмент конфига provisioner (обезличенный образец)
volumes:
  - type: bind
    source: /opt/hermes/company/skills/example-corp-context
    target: /app/skills/corp-context
    read_only: true
```

Агент при старте читает все `SKILL.md` из `/app/skills/*/SKILL.md` и
включает их в system-контекст. Пользователь не видит содержимое skill,
не может изменить его — только читать результат в поведении агента.

### Почему RO

- Пользователь не должен иметь возможность переписать корпоративный контекст
  через своего агента.
- Обновление skill — операция платформы (деплой новой версии файла),
  а не пользователя.
- Изоляция: один и тот же skill файл монтируется во все runtime — не
  дублируется на пользователя.

## Формат SKILL.md

Файл в Markdown. Обязательные секции:

```markdown
# <Название skill>

## Контекст
<Кто мы, как называется компания, чем занимаемся — кратко.>

## Сервисы
<Маппинг: как агент должен называть и использовать корп-системы.>
Пример:
- Задачи: <название трекера>, инструмент tasks-broker
- Файлы: <название файлового хранилища>, инструмент files-broker
- Каталог сотрудников: corp-dir-broker

## Поведенческие правила
<Что агент должен делать, чего не должен, как эскалировать.>

## Цитирование
<Агент обязан цитировать источник при ответах из корп-памяти.>
```

Опциональные секции: `## Термины`, `## Оргструктура`, `## Типовые сценарии`.

---

## Несколько skills

Можно монтировать несколько skills одновременно — например, общий корп-контекст
плюс специализированный для конкретного отдела:

```
/app/skills/
  corp-context/SKILL.md      ← общий для всех
  legal-dept/SKILL.md        ← только для юридического отдела
```

Provisioner монтирует нужный набор в зависимости от роли пользователя
(поле `role` из corp-dir-broker).

---

## Обновление skill

1. Внесите изменения в `SKILL.md` на хосте.
2. Перезапустите runtime пользователей, которых затрагивает skill:
   ```bash
   docker restart hermes-u-<slug>
   ```
   Или массово через admin-CLI (если настроен).
3. Новый контекст вступает в силу при следующей сессии агента.

---

## Что НЕ класть в SKILL.md

- Реальные секреты, токены, пароли — для этого есть env provisioner'а.
- Персональные данные сотрудников — для этого есть corp-dir-broker.
- Динамические данные (расписание, текущие задачи) — для этого есть брокеры
  и корп-память.

SKILL.md — статический контекст. Всё динамическое — через инструменты агента.
