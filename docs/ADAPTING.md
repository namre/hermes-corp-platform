# Адаптация под свою инфраструктуру

> Этот документ — про контракты брокеров. Метод закрытия блоков (заглушка +
> образец + готовая инструкция для LLM-ассистента) — в
> [IMPLEMENTING_BLOCKS.md](IMPLEMENTING_BLOCKS.md). Принципы — в
> [PRINCIPLES.md](PRINCIPLES.md).


Принцип: система держится на **контрактах**, а не на конкретных интеграциях.
Каждый брокер — маленький HTTP-сервис с фиксированными эндпоинтами. Замена
системы = переписывание одного `main.py` при неизменном контракте.

## Таблица замен

| Заглушка | Что заменить | Где править |
|---|---|---|
| corp-dir-broker (JSON) | HRIS / LDAP / Active Directory / HR-портал | `load_directory()` |
| files-broker (in-memory) | Yandex 360 / Google Drive / SharePoint / S3 | upload / link / freebusy |
| tasks-broker (in-memory) | Kaiten / Jira / YouTrack / Asana | доступ к данным задач |
| memory (контракт) | Hindsight / совместимый сервис памяти | `memory/README.md` |
| llm-gateway (`.env`) | LiteLLM / OpenRouter / vLLM | `LLM_BASE_URL`, `LLM_API_KEY` |

## Как написать свой брокер

1. Возьмите соответствующую заглушку как каркас.
2. Оставьте **те же эндпоинты и формат ответа** — на них опираются proxy и агент.
3. Замените тело функций доступа к данным на вызов вашего API/БД.
4. Держите доступ за `BROKER_INTERNAL_AUTH` и слушайте только внутреннюю сеть.
5. Кэшируйте тяжёлые источники (каталог сотрудников в проде обновлялся по
   расписанию, а не на каждый запрос).

### Пример: реальный каталог вместо JSON

```python
def load_directory() -> list[dict]:
    # было: чтение sample_directory.json
    # стало: запрос к вашему HR-API (с кэшем!)
    resp = httpx.get(MY_HR_API, headers={"Authorization": f"Bearer {MY_HR_TOKEN}"})
    return [normalize(u) for u in resp.json()["users"]]
```

`normalize()` приводит ответ вашей системы к полям контракта
(`email`, `full_name`, `position`, `org_unit`, `telegram_id`, `manager_email`).
Остальная платформа изменений не заметит.

## Принцип «от имени пользователя»

Брокеры в шаблоне ходят в моки общим секретом. В проде для доступа к данным
конкретного человека (его файлы, его задачи) используйте **его** OAuth-токен,
а не супер-аккаунт платформы. Хранение пер-пользовательских токенов — отдельный
key-broker (postgres) в реальной системе; в шаблон не включён намеренно, чтобы
не плодить заглушки, но в архитектуре место под него есть.
