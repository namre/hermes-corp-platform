# Модель безопасности

Главная цель — корпоративное развёртывание ИИ **с защитой персональных и
коммерческих данных**. Ниже — из чего складывается защита и, честно, чего она
пока не покрывает.

## 1. Изоляция между пользователями

Каждый сотрудник работает в отдельном контейнере-агенте со своей памятью и
своими токенами. Это процессная изоляция, а не разграничение по полю в общей
базе. Ни другой сотрудник, ни администратор через своего агента не прочитает
чужой личный банк памяти.

## 2. Срезание секретов платформы (secret stripping)

Секреты платформы — токен Telegram-бота, ключи LLM-провайдера, внутренний
broker-auth — нужны **обвязке**, но не нужны агенту пользователя. Поэтому при
создании user-runtime `provisioner.strip_platform_env()` вырезает их из
окружения по списку префиксов и ключей (`USER_RUNTIME_STRIP_ENV_*`).

Эффект: даже если пользователь уговорит своего агента распечатать всё
окружение целиком — там не будет ни токена бота, ни ключей провайдера.

## 3. Брокеры за внутренней авторизацией

Доступ к корп-системам идёт не напрямую из агента, а через брокеры. Брокеры
слушают только внутреннюю docker-сеть и проверяют общий секрет
`BROKER_INTERNAL_AUTH`. Агент получает данные «от имени пользователя» в объёме,
который разрешает брокер, — а не прямой доступ к корпоративному API с супер-токеном.

## 4. Защита от prompt-инъекций — слоями

Единой «серебряной пули» нет, поэтому защита эшелонирована:

1. **Детектор в proxy** — ловит явные паттерны вытягивания секретов и
   джейлбрейка (`security_detect()`); при срабатывании запрос блокируется.
2. **Передача чужого ввода как данных, а не инструкций** — если агент A
   передаёт что-то агенту B (сценарий A2A), payload уходит как данные в
   system-контекст, а не как команда.
3. **Предупреждение в интерфейсе** — подозрительный запрос подсвечивается
   получателю.
4. **Аудит** — все межагентные и привилегированные операции логируются.

## 5. Публичный scope с жёстким потолком

Часть запросов безопасно отдавать автоматически («коллега сейчас занят?»). Для
этого есть узкий набор «публичных» интентов, и ответ на них ограничен в **коде**,
а не в настройке: на «занят?» возвращается только `busy: bool` и время
следующего слота — без названий встреч, участников и описаний. Расширить можно
только через ревью и изменение кода — это противоядие от расползания «публичного».

## Чего модель пока НЕ покрывает (честно)

- **Контекстный toolset под конкретный вызов.** Убрать у агента-получателя
  право на запись «на лету» для одного межагентного запроса текущая база агента
  не умеет — toolset общий, не пер-запросный. Это апстрим-фича базового агента.
- **Детектор инъекций — эвристика.** Регэкспы ловят явное, но не заменяют
  ревью. Их нужно дополнять под свои сценарии.
- **Заглушки не аутентифицируют пользователя по-настоящему.** В шаблоне личность
  приходит заголовком; в проде её должен подтверждать ваш канал (подписанный
  Telegram webhook, SSO на вебе).

Перед публичным или боевым использованием — собственное security-review.
