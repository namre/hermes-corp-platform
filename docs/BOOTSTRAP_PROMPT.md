# Bootstrap-промпт: развернуть аналогичную систему с нуля

Ниже — самодостаточный промпт. Дайте его LLM-ассистенту (Claude, Cursor, Copilot),
и он соберёт мультитенантную обвязку для корпоративного ИИ-агента по принципам
этого репозитория. Можно дать как текст или ссылкой на raw-версию этого файла:
`https://raw.githubusercontent.com/namre/hermes-corp-platform/main/docs/BOOTSTRAP_PROMPT.md`

Образцы кода и контрактов — в этом же репозитории (`services/*`, `docs/*`).

## ПРОМПТ (копируйте отсюда)

Ты — инженер, который с нуля собирает **self-hosted мультитенантную обвязку для
корпоративного ИИ-агента**. Базовый агент — однопользовательский (например,
NousResearch/hermes-agent или любой OpenAI-совместимый агент-рантайм). Твоя задача —
обернуть его так, чтобы им могла безопасно пользоваться вся компания, не смешивая
данные людей и не выпуская их в чужое облако.

### Архитектура, которую нужно построить

Каналы входа (OpenWebUI / Telegram) → identity-proxy → персональные runtime-контейнеры
(по одному на сотрудника) → брокеры к корпоративным системам + внешняя память +
единый LLM-шлюз. Плюс admin-panel оператору.

### 8 принципов (не нарушать)

1. **Контейнер на пользователя**, а не один бот на всех: у каждого свой рантайм со
   своей памятью, конфигом и токенами. Изоляция процессная, не «фильтр по user_id».
2. **Брокер — единственная дверь** к корпоративной системе: маленький HTTP-сервис с
   фиксированным контрактом; замена системы не трогает агента.
3. **Доступ от имени пользователя** (его OAuth-токен), а не общим супер-ключом.
   Токены — зашифрованы at-rest.
4. **Least privilege + режимы read/write**; где провайдер не даёт раздельных scope —
   запрет мутаций форсить в коде.
5. **Секреты платформы не попадают в runtime пользователя**: при создании рантайма
   вырезай из окружения токены ботов, ключи провайдеров, внутренние секреты.
6. **Память двухконтурная**: личный банк (auto-retain/recall, изолирован) +
   корпоративный read-only банк, где агент обязан цитировать источник.
7. **Единый LLM-шлюз** (OpenAI-совместимый): смена модели/провайдера без правки агентов;
   виртуальные ключи на пользователя.
8. **Личность из канала, не из доверия к заголовку**: OpenWebUI пробрасывает email
   вошедшего, Telegram — подписанный webhook; proxy резолвит через каталог сотрудников.

### Что построить по шагам

1. **corp-dir-broker** (каталог сотрудников). Эндпоинты: `/health`,
   `/users/by-email`, `/users/by-telegram`, `/users/search`, `/org-structure`.
   Поля: email, full_name, position, org_unit, telegram_id, manager_email. Кэш с
   периодическим обновлением. Старт — на JSON-заглушке, потом твой HRIS/LDAP/AD.
2. **provisioner**: создаёт/останавливает контейнер `hermes-u-<slug>` на пользователя.
   Реализуй `strip_platform_env()` (срез секретов) и инъекцию per-user `config.yaml`
   (режимы copy/mount), режимы auth (none/copy/mount, дефолт none).
3. **identity-proxy**: вход `/v1/chat/completions`; резолв личности через corp-dir;
   `ensure runtime` через provisioner; детектор подозрительных паттернов; форвард в
   рантайм. Плюс файловые эндпоинты `/v1/files`, `/v1/files/register-existing`,
   `/v1/files/{id}/content`. Читай заголовки `X-Telegram-Id` и `X-OpenWebUI-User-Email`.
4. **key-broker**: `/v1/keys` (Bearer system-secret) → per-user виртуальный ключ к
   LLM-шлюзу, зашифрованный at-rest (pgcrypto/Vault).
5. **Брокеры files / tasks**: per-user OAuth, набор MCP-инструментов
   (files: upload/link/freebusy; tasks: my_tasks/find/create/comment/move через
   two-step commit). Старт — заглушки, потом Yandex360/GDrive и Kaiten/Jira.
6. **llm-gateway**: OpenAI-совместимый прокси (LiteLLM): model_list, fallback,
   master_key и ключи провайдеров — только через env.
7. **Память**: адаптер к внешнему сервису (контракт `retain`/`recall`); личный банк
   `user:<id>`, корп-банк `corp:<org>` read-only; конвейер наполнения из базы знаний.
8. **Каналы**: OpenWebUI (`ENABLE_FORWARD_USER_INFO_HEADERS=true`) и telegram-gateway
   (один бот → много пользователей через per-user fake-токен HMAC; webhook с секретом
   в пути; дедуп по update_id; fail-closed для неизвестного tg_id).
9. **admin-panel**: веб-UI оператору (пользователи, система, каналы, интеграции, аудит);
   данные пользователей из corp-dir, остальное — метрики платформы.
10. **Эксплуатация**: бэкап per-user рантаймов (whitelist), idle-shutdown, health/doctor,
    runbook обновления/отката (pin образа по digest), схемы БД (registry, key-broker).

### Технические требования

- Docker Compose; все порты на `127.0.0.1`, наружу только reverse-proxy с TLS.
- Брокеры доступны только во внутренней сети, проверяют общий `BROKER_INTERNAL_AUTH`.
- Никаких секретов в коде — только через env; дать `.env.example` с плейсхолдерами.
- Топология должна подниматься на заглушках без единого реального секрета.

### Критерии приёмки

- `docker compose up -d` поднимает всё на моках; `/health` всех сервисов отвечает.
- Запрос от имени пользователя резолвится в его персональный рантайм.
- Секреты платформы отсутствуют в окружении user-runtime.
- Память: личные банки изолированы; корп-банк read-only; ответы со ссылкой на источник.
- Каждый корпоративный блок заменяется без правки остальной системы.

Сначала покажи план и список сервисов с портами и env-переменными. Потом собирай
по одному сервису, начиная с corp-dir-broker (от него зависит резолв личности).
