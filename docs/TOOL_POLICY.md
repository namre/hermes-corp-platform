# Политика инструментов пользователя

У обычного пользователя нет доступа к инструментам, способным навредить
другим пользователям или платформе. Это не переключатель в конфиге — это
граница контейнера.

---

## 1. Почему не простой toggle

Наивный подход: «выключим tool X в конфиге агента». Проблема — конфиг агента
живёт внутри user-runtime. Пользователь может попросить своего агента изменить
конфиг, перезагрузить его или обойти флаг через prompt-инъекцию. Toggle в конфиге
агента — это политика доверия к агенту, не к платформе.

Правильный подход: **агент физически не может вызвать то, к чему у контейнера
нет доступа**. Если у контейнера нет сетевого маршрута до API — он туда не
дойдёт. Если у контейнера нет volume-mount с admin-данными — он их не прочитает.

---

## 2. Что разрешено обычному пользователю

| Инструмент / действие | Разрешено | Реализация границы |
|---|---|---|
| recall из личного банка памяти | ДА | runtime-токен, изолированный неймспейс |
| retain в личный банк памяти | ДА | runtime-токен |
| recall из корп-банка (read-only) | ДА | только `recall` в tools.include |
| вызов corp-dir-broker (чтение) | ДА | брокер проверяет BROKER_INTERNAL_AUTH |
| вызов files-broker (от имени пользователя) | ДА | OAuth-токен пользователя |
| вызов tasks-broker (от имени пользователя) | ДА | OAuth-токен пользователя |
| LLM-запросы через llm-gateway | ДА | виртуальный ключ пользователя |
| status_check к другому агенту (public scope) | ДА | после consent (см. A2A.md) |

## 3. Что запрещено обычному пользователю

| Инструмент / действие | Запрещено | Почему / как enforced |
|---|---|---|
| retain в корп-банк памяти | ДА | tools.include только recall + банк отклоняет retain по токену |
| доступ к памяти другого пользователя | ДА | изолированный неймспейс банка; чужой банк не существует в контексте runtime |
| прямые вызовы корп-API минуя брокер | ДА | нет сетевого маршрута из user-runtime к корп-системам напрямую |
| чтение платформенных секретов | ДА | `strip_platform_env` в provisioner; секретов нет в окружении runtime |
| управление другими user-runtime | ДА | provisioner слушает только внутреннюю сеть, не из user-runtime |
| запись в audit-лог напрямую | ДА | audit-service недоступен из user-runtime; записи идут через proxy |
| A2A-вызов без consent и вне PUBLIC_INTENTS | ДА | consent-поток + enum PUBLIC_INTENTS в коде |
| изменение своего toolset «на лету» | частично | конфиг монтируется RO; но scoped toolset per-request — известное ограничение (см. A2A.md §5) |

## 4. Enforcement — слои

### Слой 1 — граница контейнера (основной)

Docker network policy: user-runtime находится в изолированной сети. Маршруты
к корп-системам проходят только через брокеры. Брокеры проверяют
`BROKER_INTERNAL_AUTH` и не принимают запросы, не несущие валидный секрет.

```
user-runtime
  ├── corp-dir-broker   ← разрешено (BROKER_INTERNAL_AUTH)
  ├── files-broker      ← разрешено (BROKER_INTERNAL_AUTH + user-token)
  ├── tasks-broker      ← разрешено (BROKER_INTERNAL_AUTH + user-token)
  ├── llm-gateway       ← разрешено (virtual-key пользователя)
  ├── provisioner       ← нет маршрута
  ├── audit-service     ← нет маршрута (только через proxy)
  └── [корп-API]        ← нет маршрута (только через брокеры)
```

### Слой 2 — конфиг агента (дополнительный)

tools.include/exclude и banking-политика задаются при создании runtime через
provisioner и монтируются read-only. Пользователь не может изменить файл
конфигурации внутри контейнера — volume примонтирован RO.

### Слой 3 — детектор в proxy (эвристический)

identity-proxy проверяет входящие запросы на явные паттерны превышения
полномочий. При срабатывании — блокирует запрос и записывает security-событие
в audit-лог.

### Слой 4 — аудит

Все tool-call в привилегированных и A2A-сессиях логируются. Аудит позволяет
обнаружить попытки обхода post-factum.

## 5. Конфигурация tools для user-runtime (образец)

```yaml
# config.yaml — монтируется RO в user-runtime
tools:
  memory:
    banks:
      - name: "user:__current__"
        include: ["recall", "retain"]
      - name: "corp:myorg"
        include: ["recall"]   # только чтение; retain отсутствует намеренно

  brokers:
    corp_dir:
      mode: "read"            # только чтение каталога
    files:
      mode: "read_write"      # от имени пользователя
    tasks:
      mode: "read_write"      # от имени пользователя

  a2a:
    outbound_public_intents:
      - "status_check"        # PUBLIC_INTENTS enum; расширять только через код

# Платформенные секреты (strip_platform_env — их здесь нет)
# BROKER_INTERNAL_AUTH — не попадает в user-runtime
# TELEGRAM_BOT_TOKEN  — не попадает в user-runtime
# LLM_PROVIDER_KEY    — не попадает в user-runtime
```

## 6. Известные ограничения (честно)

- **Scoped toolset per-request** — при A2A-вызове нельзя ограничить toolset
  только на время этого вызова. Это upstream-фича Hermes-Agent, пока не
  реализована. Текущий workaround — prompt + детектор + аудит (см. A2A.md §5).
- **Детектор в proxy — эвристика.** Ловит явные паттерны, не всё.
  Дополняйте regex под свои сценарии.
- **Volume RO не защищает от изменений внутри контейнера в tmpfs.** Если агент
  создаст конфиг в памяти (tmpfs) и перезапустится — он прочитает RO-файл.
  Вектор не критичен, но знать стоит.
