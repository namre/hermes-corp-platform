# Принципы построения

Это главный документ репозитория. Конкретные интеграции (Yandex 360, Kaiten,
LiteLLM, Hindsight) у каждого свои — а принципы одни. Если поймёте их, соберёте
обвязку под любой стек.

## 1. Контейнер на пользователя, а не один бот на всех

Базовый агент однопользовательский. Чтобы дать его компании и не смешать данные,
каждому сотруднику поднимается **отдельный runtime-контейнер** со своей памятью,
конфигурацией и токенами. Изоляция процессная, а не «фильтр по user_id». Чужую
память нельзя прочитать в принципе — её нет в этом процессе.

## 2. Брокер — единственная дверь к корпоративной системе

Агент не ходит в корпоративный API напрямую. Между ними — **брокер**: маленький
HTTP-сервис с фиксированным контрактом. Это даёт:
- замену системы без переписывания агента (меняется тело брокера, контракт тот же);
- единую точку аудита и ограничения прав;
- возможность отдать в опенсорс контракт, оставив реализацию приватной.

## 3. Доступ — от имени пользователя, а не супер-аккаунтом

Брокер ходит в данные конкретного человека по **его** токену (OAuth), а не общим
admin-ключом. Поэтому агент видит ровно то, что видит сам сотрудник. Токены
хранятся **зашифрованными at-rest** (мастер-ключ в окружении брокера), привязка —
по email пользователя.

## 4. Least privilege и режимы read/write

Каждая интеграция различает режимы. На чтение запрашивается минимальный scope
(например, IMAP-read у почты), на запись — отдельный. Где провайдер не даёт
раздельных scope (у Yandex Calendar только `calendar:all`), ограничение на запись
**форсится на уровне брокера**: в read-режиме брокер отклоняет мутации.

## 5. Секреты платформы не попадают в runtime пользователя

Токен бота, ключи LLM-провайдера, внутренний broker-auth нужны обвязке, но не
агенту пользователя. При создании runtime provisioner **вырезает** их из
окружения (`strip_platform_env`). Даже если пользователь уговорит агента
распечатать всё окружение — секретов там не будет.

## 6. Память: личная (rw) + корпоративная (ro со ссылками)

Два контура. Личный банк `user:<id>` — auto-retain/recall, изолирован. Корпоративный
`corp:<org>` — только чтение, наполняется отдельным конвейером, агент обязан
**цитировать источник** (документ, дата, версия), а не пересказывать по памяти.

## 7. Единая точка к моделям

Агент ходит к моделям через один OpenAI-совместимый шлюз (LiteLLM или аналог).
Это даёт смену провайдера и модели без правки агентов, виртуальные ключи на
сервис, лимиты и фолбэк.

## 8. Личность приходит из канала, а не из доверия к заголовку

Кто пользователь — определяет канал входа: OpenWebUI пробрасывает email
вошедшего, Telegram — подписанный webhook. proxy резолвит этот идентификатор
через каталог сотрудников. Голому заголовку из публичной сети доверять нельзя —
его подтверждает аутентификация канала.
