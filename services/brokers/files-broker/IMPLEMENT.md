Ты реализуешь **files-broker** для корпоративной агентской платформы. Сейчас это
заглушка на моках (`main.py`). Замени её на реальную интеграцию, СОХРАНИВ контракт.

## Моя система
- Облако/почта/календарь: <Yandex 360 | Google Workspace | Microsoft 365 | другое>
- OAuth client_id берётся из: <кабинет провайдера>
- Нужные сервисы: <disk, calendar, mail, telemost…>
- Внутренний домен/redirect URI: <https://agent.mycorp.local/oauth/callback>

## Контракт (не менять)
- `/health`, `/api/oauth/{init,exchange,status,revoke}`, `/api/preferences`, `/mcp`.
- Доступ только из внутренней сети, проверка заголовка `BROKER_INTERNAL_AUTH`.
- Личность пользователя приходит как email в заголовке от proxy.

## Обязательные инварианты
1. Доступ к данным — по **OAuth-токену пользователя**, не общим ключом.
2. Токены — **зашифрованы at-rest** мастер-ключом из env (`FILES_TOKEN_MASTER_KEY`).
3. Режимы **read/write** на каждый сервис, least privilege. Где провайдер не даёт
   раздельных scope — запрет мутаций форсить в коде для read-режима.
4. Все секреты — только через env, ничего не хардкодить.

## Что реализовать
- OAuth-флоу (init→consent→exchange), хранение зашифрованных токенов по email.
- MCP-инструменты из `TOOLS.md` для выбранных сервисов.
- `make_public_link` для disk (в заглушке фейковая — сделать реальный шэр в
  облако пользователя; в Yandex это отдельная операция, проверь её наличие в API).
